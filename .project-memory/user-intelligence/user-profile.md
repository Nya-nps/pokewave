# Profil Utilisateur — Nya-nps

Dernière mise à jour : 2026-06-09
Sessions analysées : 1

---

## Identité projet
- **GitHub** : Nya-nps
- **Projet** : PokéQuest (RPG Pokémon navigateur, vanilla JS)
- **Rôle** : Développeur solo / game designer

---

## Communication
- **Langue** : Français exclusivement
- **Style** : Direct, instructions structurées en listes et blocs
- **Niveau de détail** : Élevé — préfère des systèmes complets, pas des ébauches
- **Ton préféré** : Professionnel, sans superflu, sans emojis
- **Format des demandes** : Souvent en missions formelles avec objectifs, règles et livrables explicites
- **Longueur des réponses attendues** : Concises sur les explications, exhaustives sur les livrables

---

## Profil technique
- **Niveau** : Avancé — comprend les implications de conflits git, de l'ordre de chargement JS, des monkey-patches
- **Stack** : Vanilla JS, HTML, CSS — pas de frameworks, pas de build tools
- **Environnement** : Windows 11, PowerShell
- **Git** : Utilise git activement (commits fréquents, messages structurés feat/fix/balance/refactor)

---

## Style de travail
- **Organisation** : Très structuré — crée des systèmes de mémoire, des index, des docs internes
- **Efficacité** : Priorité absolue à la réduction de tokens et de lectures inutiles
- **Parallélisme** : Conscient des risques de conflits sur fichiers partagés, prudent, valide avant d'agir
- **Qualité** : Attend des livrables complets et directement utilisables
- **Itération** : Préfère tout faire bien du premier coup plutôt que d'itérer
- **Validation** : Pose des questions courtes pour confirmer avant d'agir

---

## Objectifs récurrents détectés
1. Minimiser la consommation de tokens par session
2. Éviter les relectures complètes de fichiers volumineux
3. Construire des systèmes persistants et réutilisables
4. Maintenir un équilibre gameplay précis (balancing)
5. Garder le code performant (FPS, lag)
6. Travailler de manière organisée et documentée

---

## Ce qu'il NE faut PAS faire
- Ne jamais lire game.js en entier (131k tokens)
- Ne jamais sur-expliquer des notions techniques
- Ne jamais livrer des prototypes ou ébauches
- Ne jamais ajouter de features non demandées
- Ne jamais scanner tout le dépôt sans raison
